# 2026-07-21

Direction day — no code. Decided the **v2 repositioning**: BrainCue widens from
interview copilot to **universal ambient conversational companion** — it sits in
your real conversations and activities (interviews on either side, meetings,
tutoring, ambient companionship while working/gaming) and contributes through
the overlay or voice. Interviews become one mode of a shared engine.

**Why this framing (and not "voice mode clone"):** generic voice chat is
ChatGPT/Gemini turf. BrainCue's structural moat is what a cloud chat app can't
do — loopback presence inside real calls, a capture-invisible overlay, local
grounding in the user's corpus, BYO key. Positioning: "the AI in the room with
you"; voice is a surface, not the product.

**Key insight from the code audit:** the universal primitives largely exist —
realtime STT, question classifier (= a reactive trigger policy), RAG, streaming
answers, TTS, and a full two-way voice loop in sparring/mock. The pivot is
engine extraction + repositioning, not a rebuild.

## Decisions

- Keep the **BrainCue** name and appId; retire the "interview copilot"
  descriptor. "Cue Card" stays.
- Interview mode is the regression gate — no refactor may degrade it.
- Companion/game-buddy is deliberately **last** (needs memory, interjection
  policy, realtime voice, cost governance).
- Mock/sparring migrate onto the engine's dialogue loop **in Phase 3** (with
  Tutor), not during the Phase 1 extraction.

## Prompt 2 — Space/ContextPack vocabulary + schema (`feat/v2-schema-packs`)

**Design call (deviation from the 12-plan sketch): the rename is LOGICAL, not
physical.** Drizzle maps TS names to legacy SQL names (`contextPacks` →
table `jobs`, `packId` → column `job_id`), so migration 0008 is just three
ALTERs + one UPDATE — no table rebuild, no FK re-pointing, no fkRebuild.ts
changes, and `db:generate` stayed non-interactive (drizzle-kit prompts on
physical renames).

- **0008** (`drizzle/0008_flaky_venom.sql`): +`jobs.kind` (default 'job'),
  +`sessions.mode` (default 'interview'), −`profiles.answer_style` (dead),
  + backfill `mode = kind='live' ? 'interview' : 'practice'`.
- **Backup-before-migrate** (`db/index.ts`): pending migrations detected via
  journal vs `__drizzle_migrations` count → `VACUUM INTO app.db.pre-migrate.bak`
  before `migrate()`. The rollback story the runner lacked.
- **Shared types**: `SessionMode`, `ContextPackKind`, `ContributionKind/Status`
  + `Contribution` model, `ContextPack` (+`Job` alias), `Session.mode`,
  Speaker widened to `you|them|agent|unknown` ∪ legacy + `normalizeSpeaker()` —
  **no write-path or row changes**, overlay pixel-identical.
- **Repos**: jobs.repo.ts is now `contextPacksRepo` (+deprecated `jobsRepo`
  alias); sessions/applications/mock/sparring/sessionManager map `packId` ↔ the
  IPC-compat `jobId` field; mock/sparring inserts write `mode:'practice'`.
- **Zod enums single-sourced** in `ipc/schemas.ts` (were duplicated inline in
  session/mock/sparring/profiles .ipc.ts) + v2 enums ready (`zSessionMode`,
  `zContextPackKind`). No channel changes; Session payloads gain `mode`.
- **Lossless-migration proof**: committed real v1.5.x fixture
  (`test/fixtures/pre-v2.db`, generated from migrations 0000–0007 with every

## Prompt 3 — conversation-engine extraction (`feat/v2-engine`)

Behavior-preserving extraction of the live pipeline out of `sessionManager.ts`
(672 lines → ~130-line facade). **All 8 parity tests passed against the engine
UNMODIFIED** — the proof the extraction changed nothing: event order,
persistence, the classify gate, stop/cleanup, and the stale-abort slot rules
are byte-identical (comments transplanted with the code).

- `services/engine/`: `engine.ts` (singleton: one active session, lifecycle,
  sources) + `engineSession.ts` (generic flow: ContextEvent → trigger →
  grounding → generation → contribution → persistence; owns the
  answering/answerAbort ownership rules) + `contextEvent.ts`,
  `modeDefinition.ts`, `grounding.ts` (top-k=5), `sourceAdapter.ts`,
  `trigger/{triggerPolicy,reactiveQuestionPolicy(0.4 floor),summonedPolicy}`,
  `modes/interview.mode.ts` (persona=streamAnswer, follow-up prediction,
  legacy 'interviewer' speaker label for parity), `persistence/`.
- **Migration 0009**: generic `contributions` table. Completed answers
  dual-write (ai_answers stays the parity source of truth) with provenance
  sourceRefs (question + chunk ids). Same only-on-completion semantics —
  aborted streams write nothing.
- `sessionManager.ts` = thin facade, every v1 signature intact — zero call
  sites changed (IPC, mock/sparring, coding solver, report).
- Module-level singletons became session-scoped: `followupGeneration` map and
  the audio-level throttle live on EngineSession; "session replaced" checks
  went from `live?.sessionId !== id` to `this.stopped` (set on teardown) —
  same semantics, now ownable per session.
- One deliberate tightening: `answerQuestion` on a session row that exists but
  is NOT live now throws 'Session is not live' (v1 would half-run with
  fallback prefs and no slot owner — an unreachable path for all real callers:
  mock targets the live session, ask/regenerate require one).
- New tests: trigger-policy units (incl. the ≥0.4 boundary), direct-ask
  bypasses the classifier, pause-wins, contribution dual-write + provenance,
  aborted-stream-writes-no-contribution (added to the stale-abort race).
- Docs: 02-ARCHITECTURE §4 rewritten around the engine; 05/06 untouched this
  PR (no IPC change; OpenAI call sites unchanged — the provider seam cuts at
  interview.mode.ts next).
- Not run: the privacy hard test (needs a display + packaged app; this PR
  touches no windows/IPC/renderer code — run it at the phase boundary).
