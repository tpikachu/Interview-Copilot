## Cue Card answer controls — Length / Format / Pronunciation + Clear & Regenerate

Reworked answer styling into two orthogonal axes and added live Cue Card controls.

**Design decision (with the user):** the old `answerStyle` (`concise | detailed |
star | technical | conversational`) mixed two concerns — *length* and *tone/format*.
Split them:

- `AnswerStyle` is now **format/tone** only: `default | star | technical | conversational`.
- New `AnswerLength` = `key_points | detailed` — the live Cue Card toggle the user
  asked for (short, key-point-focused but natural / very detailed but natural).
- New `pronunciation` boolean — annotate rare/technical/foreign terms with a quick
  phonetic respelling the first time they appear.

`answerStyle` is runtime-only (not persisted on sessions), so no migration. The
`profiles.answer_style` column is persisted but vestigial; legacy `concise`/`detailed`
values are normalized to `default` at read (`profiles.repo.toAnswerStyle`) and on the
profiles IPC (zod `.transform`).

### How it flows

- Setup (`SessionPage`) seeds Format + Length for the round → `session.start/resume`
  → `live.answerStyle` / `live.answerLength`. Pronunciation starts off.
- `answer.ts` builds the prompt from explicit LENGTH + FORMAT + (optional)
  pronunciation instructions instead of a single "Desired style" line.
- Live controls act on the **active** session (no sessionId from the overlay,
  mirroring `togglePauseActive`):
  - `session:set-answer-prefs` `{ style?, length?, pronunciation? }` → updates `live`.
  - `session:regenerate` → re-runs the **last question** (stored as `live.lastQuestion`).
  - `session:clear-answer` → aborts the in-flight stream (`live.answerAbort`, an
    `AbortController` now passed into `streamAnswer`); the overlay also clears locally.
- `goLive` broadcasts `session:answer-prefs` (`EVENTS.answerPrefs`) to the overlay so
  its toggles start in sync with the round's setup choice.

### Overlay (Cue Card) UI

New answer-controls row above the answer: a Key points / Detailed segmented toggle,
a pronunciation toggle (æ), Regenerate (↻), and Clear (🗑). Changing length or
pronunciation updates prefs and auto-regenerates the current question so the new
form appears immediately. Row is `data-ct-interactive` + `no-drag` so it stays
clickable under click-through.

## Move the live transcript + controls into the Cue Card

The dashboard used to swap to a big live view (transcript + pause/stop) on
start/resume — useless once you minimize the dashboard during an interview. Moved
the live surface into the overlay.

**Overlay (`Overlay.tsx`) is now the full live surface:**
- Subscribes to `EVENTS.transcriptDelta` and keeps its own `transcript[]` / `interim`
  / `lineId` (mirrors the dashboard store's reducer); appends a 'detected question'
  line on `questionDetected`. Resets all of it on a fresh live transition (a
  `prevLive` ref guards against resetting on pause/resume).
- New scrollable **transcript pane** (`flex-[35]`, `min-h-0`, jump-to-latest) above
  the **answer** (`flex-[65]`) — ~35/65 split of the flexible area.
- Controls now include a **Format** select (in addition to Length / pronunciation /
  regenerate / clear) and a **Stop** button (shown only when live).

**Stop from the Cue Card:** new `session:stop-active` IPC → `sessionManager.stopActive()`
wraps `stop(live.sessionId)`. Since the overlay bypasses the dashboard store's
`stop()`, the store's `onSessionState` now tears down (stopCapture + clear session)
when `status === 'stopped'` — so a Cue Card Stop also drops the dashboard view and
releases the mic. `stopCapture()` is idempotent, so the dashboard's own stop calling
both paths is fine.

**Dashboard (`SessionPage.tsx`)** no longer shows the big live view: on going live it
shows a compact bar (status badges + Pause + Stop + waveform + mic error) and a note
that the transcript/cues/controls are in the Cue Card. Kept the manual Ask box.
Removed the now-dead transcript machinery (MAX_LINES, scrollRef, atBottom,
onTranscriptScroll, the auto-scroll effect, transcript/interim from the store).

Planning done first via a 4-agent Understand workflow (transcript flow / overlay
layout / dashboard live view / stop-pause IPC), then implemented, then an adversarial
review pass on the diff — which caught a real bug, fixed below. Typecheck + build green.

**Review fixes:**
- **(HIGH) Regenerate duplicated the question.** Toggling length/format/pronunciation
  auto-regenerates; the old `regenerateActive` called `answerQuestion`, which
  re-inserted a `detectedQuestions` row and re-broadcast `questionDetected` →
  duplicate "detected question" lines in both transcripts + duplicate DB rows that
  distort reports. Fixed by splitting `answerQuestion` (register) from a new
  `generateAnswer(sessionId, questionId, questionText)` (stream). `regenerateActive`
  now aborts, broadcasts `EVENTS.answerReset` (overlay clears the answer only), and
  re-runs `generateAnswer` reusing the SAME questionId; `generateAnswer` deletes
  prior `aiAnswers` for that question so a regenerate overwrites.
- **(MED) `stop()` over-reached.** It broadcast `stopped` + hid the overlay even when
  stopping a non-live session row. Gated all teardown on `wasLive`.
- **(MED) layout squeeze.** Added `shrink-0` to the overlay header / opacity / controls
  rows so the transcript + answer panes absorb the flex.
- **(LOW) dangling interim.** Overlay clears `interim`/`streaming` on session stop.

**Verification pass** (re-reviewed the fixes) confirmed all four resolved + abort
ordering correct, and caught one regression I'd introduced: `generateAnswer` deleted
the prior answer *before* streaming, so an aborted regenerate (Clear/Stop mid-stream)
could leave a question with zero persisted answers. Fixed by moving the `DELETE` to
sit immediately before the `INSERT` (adjacent + synchronous), after the stream
completes — an aborted regenerate now returns before either runs, preserving the
existing answer. Also added `live.answerAbort?.abort()` on stop so stopping mid-stream
cancels the in-flight answer instead of streaming to a hidden overlay.
