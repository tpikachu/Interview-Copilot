# 2026-07-15

**Critical bug: Privacy Mode was not hiding anything.** User report: every window
visible while screen sharing, despite Privacy Mode ON.

> ⚠️ **v1.5.1's fix was WRONG and is superseded (see "Correction" below).** The
> DirectComposition theory did not hold up under rigorous testing; the real
> trigger — surfaced by the user — is **moving the window**. Kept the original
> notes for the record, then corrected.

## Diagnosis (fully reproduced + isolated on this machine)

Built a ground-truth harness: launch the built app (isolated userData, privacy ON
by default), then capture the dashboard's own screen rect from a SECOND Electron
process via `desktopCapturer` — DXGI/WGC, the same capture family Zoom/Meet/Teams
use, i.e. exactly what WDA_EXCLUDEFROMCAPTURE must defeat. (PowerShell is broken
in this environment and the sandbox blocks freshly compiled EXEs, so Electron
itself became the capture oracle.)

Findings, each verified with a screenshot:

1. Real app, privacy ON → **fully readable in capture** (bug reproduced; the app
   truthfully reported `enabled: true` the whole time).
2. Minimal Electron probe (`setContentProtection(true)` at creation AND on show)
   → **also leaked** ⇒ not our application logic.
3. Same probe on **Electron 43** → still leaked ⇒ not fixable by upgrading.
4. Same probe with `--disable-direct-composition` → **properly excluded**.
5. Control: unprotected probe under the same switch → visible ⇒ the oracle is
   valid and the switch doesn't just hide everything.

**Root cause:** on this Windows 11 build (10.0.26200) + GPU driver, Chromium's
DirectComposition presentation path renders via overlay planes that DWM's
capture-exclusion filter never sees — `SetWindowDisplayAffinity(WDA_EXCLUDE-
FROMCAPTURE)` is applied but bypassed. Matches field reports of WDA failing for
Chromium apps specifically (notepad hides, chrome.exe doesn't) on a subset of
Windows 11 machines.

## Fix

`src/main/index.ts`: on win32, `app.commandLine.appendSwitch('disable-direct-
composition')` before ready (escape hatch: `BRAINCUE_ALLOW_DIRECT_COMPOSITION=1`
for GPUs that misbehave without DC, accepting the leak). Verified end-to-end on
the rebuilt app: privacy ON → app absent from capture (background shows through);
privacy OFF → app visible. Perf cost for a text UI is negligible.

Also added `scripts/verify-privacy-capture.mjs` — the harness, kept in-repo so
this exact leak class can be re-checked in minutes after any Windows feature
update or Electron upgrade (its captures contain the real screen; delete after
inspection).

## Correction (same day): the real cause is move-drops-protection

The v1.5.1 DirectComposition fix was disproven by rigorous multi-launch testing:
`--disable-direct-composition`, `--disable-gpu`, and default rendering all leaked
at the same rate (5–6/6 launches; 15/15 within a session). What looked like a
working fix was single-shot captures landing on an intermittently-excluding state.
Also confirmed: the window/privacy code is byte-identical to v0.7.0 (the version
the user said worked) on the same Electron — so the app never regressed.

Then the user gave the decisive clue: **"working fine at first, but when I move
the window it shows again."** That's the mechanism — on Windows,
`SetWindowDisplayAffinity(WDA_EXCLUDEFROMCAPTURE)` is dropped when the window
enters the modal move/resize loop (a user drag), and **nothing in our code
re-asserted it**: the overlay only saved bounds on move; the main window only
re-applied on `show`. So after any drag the "hidden" window reappeared in capture.

Also learned (dead-ends worth recording): an in-app self-test can't work — a
process's own `desktopCapturer` does NOT see its own content-protected window
(measured: 0 px protected vs 1876 px unprotected), so the app would always read
"hidden"; only a separate process is a valid oracle, and even that is confounded
by the intermittency. So auto-detection was abandoned.

**Fix (`keepContentProtected` in privacy.ts):** re-assert `setContentProtection`
on `move` / `resize` (fire during/after a Windows drag) plus `restore` / `focus`
/ `show`, wired into all three windows (main, overlay, selector). Respects the
on/off state, so it correctly clears protection when Privacy Mode is off.
Removed the false DirectComposition switch. Verified: typecheck · 179 unit
(+4 wiring tests for the re-assertion) · build green. **The OS drag behavior
itself is only reproducible on the affected machine, so the definitive check is
the reporter dragging the real window on this build** — the in-harness real-mouse
drag couldn't be simulated and the machine's WDA state is intermittent.

## The precise trigger: FOCUS / CLICK (not just move) — verified fix

The user refined it further: "when I click the window — even a meaningless
position — it shows suddenly." Focus/activation is the real trigger (move is one
instance of it). This was scriptable, so I reproduced and A/B-verified it:

Magenta-sentinel probe (separate-process capture, the valid oracle):
- protected, unfocused → **0 px** (excluded)
- after focusing the window → **56588 px** (FULLY visible — the bug)
- after re-asserting on focus → **0 px** (hidden again, across 3 cycles)

A/B on the real built app, white backdrop + programmatic clicks:
- **OLD v1.5.1:** before-click lum=140 → after clicks **[140, 21, 21]** (21 = dark
  app UI = LEAK on click)
- **NEW (fix):** before-click lum=140 → after clicks **[140, 140, 140]** (stays
  hidden through every click)

This also explains the earlier "6/6 leak" runs: the E2E harness was focusing the
window. Everything reconciles — a static unfocused protected window is excluded;
**focus/click/move/resize drop the affinity**, and nothing re-asserted it.

Fix = `keepContentProtected` (re-assert on focus/move/resize/restore/show) +
`startContentProtectionWatchdog` (500ms backstop for activation paths that fire no
clean event: taskbar, Alt-Tab, display/DWM changes). Verified end-to-end.

## Final round: real-input leak matrix → the complete, measured picture

User: still flashing "by some event" (and the 500ms watchdog looked like
*interval* flashing, so it was removed in 0482672). Built the definitive
harness: **real OS clicks** (koffi → `mouse_event`/`SetCursorPos`, the same
low-level path as hardware — unlike `win.focus()`, these DO reproduce the
drops) + a **separate-process capture oracle** at ~55fps
(`requestVideoFrameCallback` over a WGC/DXGI `desktopCapturer` stream) with a
white click-through backdrop and per-frame leak metrics, plus positive/negative
controls each run. Scratchpad: `privharness/` (orchestrate.js / oracle-main.js /
input.js).

Measured on this machine (Win 11 26200):

1. `GetWindowDisplayAffinity` **never changes during a leak** (~1M samples,
   stays 0x11 while the window is plainly visible in capture). The drop is
   internal to DWM composition → polling can't detect it; only re-CALLING
   `SetWindowDisplayAffinity` heals it.
2. **Re-calling `setContentProtection` is invisible in capture** (0 leak frames
   across 400+ re-calls, twice) — and WDA never hides a window locally, so
   re-asserting has no visual cost anywhere. The old watchdog wasn't *causing*
   the interval flashing — it was *healing spontaneous drops at a 500ms
   cadence* (each drop sat visible up to 500ms → looked periodic).
3. Real leak classes (old build): **click on an already-active window**
   (~1-in-5, fires NO activation message → stayed visible indefinitely),
   **native `<select>` popups** (separate OS window, affinity 0 → always
   visible while open; matches the user's recording), **z-order changes**
   (`SetWindowPos`, no Electron event → visible for seconds), and **delayed
   spontaneous drops** (~470ms after a drag, no signal at all).
4. `focusable: false` was REJECTED by the data: activation isn't the (only)
   trigger — non-activating clicks drop it too, and would fire even fewer
   healing signals. No UX sacrifice needed (Ask box + dropdowns keep keyboard).
5. `settings.set({ privacyMode })` was never a real API (zod strips unknown
   keys) — the old verify script silently tested nothing. Fixed.

**Fix shipped (this branch):**
- `keepContentProtected`: synchronous re-assert inside
  WM_MOUSEACTIVATE/ACTIVATE/NCACTIVATE/SETFOCUS + **WM_PARENTNOTIFY** (clicks
  on active window) + **WM_WINDOWPOSCHANGED** (z-order) +
  **WM_NCLBUTTONDOWN/ENTER/EXITSIZEMOVE/CAPTURECHANGED** (drag start/end) +
  webContents `input-event` mouseDown, each with collapsed 30/120/300ms
  trailing taps; cleanup on `closed`.
- `startContentProtectionWatchdog(50)`: fast backstop for the spontaneous
  drops (skips when Privacy Mode is off; unref'd).
- All 6 overlay native `<select>`s → new in-window `Dropdown` (ui.tsx): the
  option list renders inside the protected window. (Dashboard pages still use
  native selects — convert opportunistically; during a share the user works
  from the Cue Card.)

**Verified on the fixed build** (46 segments, ~2000 frames, including a REAL
live session via `data.loadSamples` + `session.start`): 8s idle watchdog soak
0/447 frames; click-inactive 5/5, click-active 12/12 (one 1-frame blip),
z-order churn 5/5, drags 8/8, dashboard 3/3 clean; settings + live Type
dropdowns open **zero OS popup windows** and leak nothing; all 4 Format
buttons + toggles clean. Residual: the OS still drops occasionally, now
bounded to ~1–3 capture frames (≤ ~0.1s; ≈0–2 frames of a 30fps Meet share) —
the floor achievable app-side on this Windows build. Definitive sign-off =
user's live Meet/Zoom test.
