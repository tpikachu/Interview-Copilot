# 2026-07-15

**Critical bug: Privacy Mode was not hiding anything.** User report: every window
visible while screen sharing, despite Privacy Mode ON.

> ⚠️ **v1.5.1's fix was WRONG and is superseded (see "Correction" below).** The
> DirectComposition theory did not hold up under rigorous testing; the real
> trigger — surfaced by the user — is **moving the window**. Kept the original
> notes for the record, then corrected.

## Diagnosis (fully reproduced + isolated on this machine)

Built a ground-truth harness: launch the built app (isolated userData, privacy ON
by default), then capture the dashboard's own screen rect from a SECOND Electron
process via `desktopCapturer` — DXGI/WGC, the same capture family Zoom/Meet/Teams
use, i.e. exactly what WDA_EXCLUDEFROMCAPTURE must defeat. (PowerShell is broken
in this environment and the sandbox blocks freshly compiled EXEs, so Electron
itself became the capture oracle.)

Findings, each verified with a screenshot:

1. Real app, privacy ON → **fully readable in capture** (bug reproduced; the app
   truthfully reported `enabled: true` the whole time).
2. Minimal Electron probe (`setContentProtection(true)` at creation AND on show)
   → **also leaked** ⇒ not our application logic.
3. Same probe on **Electron 43** → still leaked ⇒ not fixable by upgrading.
4. Same probe with `--disable-direct-composition` → **properly excluded**.
5. Control: unprotected probe under the same switch → visible ⇒ the oracle is
   valid and the switch doesn't just hide everything.

**Root cause:** on this Windows 11 build (10.0.26200) + GPU driver, Chromium's
DirectComposition presentation path renders via overlay planes that DWM's
capture-exclusion filter never sees — `SetWindowDisplayAffinity(WDA_EXCLUDE-
FROMCAPTURE)` is applied but bypassed. Matches field reports of WDA failing for
Chromium apps specifically (notepad hides, chrome.exe doesn't) on a subset of
Windows 11 machines.

## Fix

`src/main/index.ts`: on win32, `app.commandLine.appendSwitch('disable-direct-
composition')` before ready (escape hatch: `BRAINCUE_ALLOW_DIRECT_COMPOSITION=1`
for GPUs that misbehave without DC, accepting the leak). Verified end-to-end on
the rebuilt app: privacy ON → app absent from capture (background shows through);
privacy OFF → app visible. Perf cost for a text UI is negligible.

Also added `scripts/verify-privacy-capture.mjs` — the harness, kept in-repo so
this exact leak class can be re-checked in minutes after any Windows feature
update or Electron upgrade (its captures contain the real screen; delete after
inspection).

## Correction (same day): the real cause is move-drops-protection

The v1.5.1 DirectComposition fix was disproven by rigorous multi-launch testing:
`--disable-direct-composition`, `--disable-gpu`, and default rendering all leaked
at the same rate (5–6/6 launches; 15/15 within a session). What looked like a
working fix was single-shot captures landing on an intermittently-excluding state.
Also confirmed: the window/privacy code is byte-identical to v0.7.0 (the version
the user said worked) on the same Electron — so the app never regressed.

Then the user gave the decisive clue: **"working fine at first, but when I move
the window it shows again."** That's the mechanism — on Windows,
`SetWindowDisplayAffinity(WDA_EXCLUDEFROMCAPTURE)` is dropped when the window
enters the modal move/resize loop (a user drag), and **nothing in our code
re-asserted it**: the overlay only saved bounds on move; the main window only
re-applied on `show`. So after any drag the "hidden" window reappeared in capture.

Also learned (dead-ends worth recording): an in-app self-test can't work — a
process's own `desktopCapturer` does NOT see its own content-protected window
(measured: 0 px protected vs 1876 px unprotected), so the app would always read
"hidden"; only a separate process is a valid oracle, and even that is confounded
by the intermittency. So auto-detection was abandoned.

**Fix (`keepContentProtected` in privacy.ts):** re-assert `setContentProtection`
on `move` / `resize` (fire during/after a Windows drag) plus `restore` / `focus`
/ `show`, wired into all three windows (main, overlay, selector). Respects the
on/off state, so it correctly clears protection when Privacy Mode is off.
Removed the false DirectComposition switch. Verified: typecheck · 179 unit
(+4 wiring tests for the re-assertion) · build green. **The OS drag behavior
itself is only reproducible on the affected machine, so the definitive check is
the reporter dragging the real window on this build** — the in-harness real-mouse
drag couldn't be simulated and the machine's WDA state is intermittent.

Ships as **v1.5.2** (correcting v1.5.1) once drag-tested.
