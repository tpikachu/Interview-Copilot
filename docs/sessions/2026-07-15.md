# 2026-07-15

**Critical bug: Privacy Mode was not hiding anything.** User report: every window
visible while screen sharing, despite Privacy Mode ON.

> ⚠️ **v1.5.1's fix was WRONG and is superseded (see "Correction" below).** The
> DirectComposition theory did not hold up under rigorous testing; the real
> trigger — surfaced by the user — is **moving the window**. Kept the original
> notes for the record, then corrected.

## Diagnosis (fully reproduced + isolated on this machine)

Built a ground-truth harness: launch the built app (isolated userData, privacy ON
by default), then capture the dashboard's own screen rect from a SECOND Electron
process via `desktopCapturer` — DXGI/WGC, the same capture family Zoom/Meet/Teams
use, i.e. exactly what WDA_EXCLUDEFROMCAPTURE must defeat. (PowerShell is broken
in this environment and the sandbox blocks freshly compiled EXEs, so Electron
itself became the capture oracle.)

Findings, each verified with a screenshot:

1. Real app, privacy ON → **fully readable in capture** (bug reproduced; the app
   truthfully reported `enabled: true` the whole time).
2. Minimal Electron probe (`setContentProtection(true)` at creation AND on show)
   → **also leaked** ⇒ not our application logic.
3. Same probe on **Electron 43** → still leaked ⇒ not fixable by upgrading.
4. Same probe with `--disable-direct-composition` → **properly excluded**.
5. Control: unprotected probe under the same switch → visible ⇒ the oracle is
   valid and the switch doesn't just hide everything.

**Root cause:** on this Windows 11 build (10.0.26200) + GPU driver, Chromium's
DirectComposition presentation path renders via overlay planes that DWM's
capture-exclusion filter never sees — `SetWindowDisplayAffinity(WDA_EXCLUDE-
FROMCAPTURE)` is applied but bypassed. Matches field reports of WDA failing for
Chromium apps specifically (notepad hides, chrome.exe doesn't) on a subset of
Windows 11 machines.

## Fix

`src/main/index.ts`: on win32, `app.commandLine.appendSwitch('disable-direct-
composition')` before ready (escape hatch: `BRAINCUE_ALLOW_DIRECT_COMPOSITION=1`
for GPUs that misbehave without DC, accepting the leak). Verified end-to-end on
the rebuilt app: privacy ON → app absent from capture (background shows through);
privacy OFF → app visible. Perf cost for a text UI is negligible.

Also added `scripts/verify-privacy-capture.mjs` — the harness, kept in-repo so
this exact leak class can be re-checked in minutes after any Windows feature
update or Electron upgrade (its captures contain the real screen; delete after
inspection).

## Correction (same day): the real cause is move-drops-protection

The v1.5.1 DirectComposition fix was disproven by rigorous multi-launch testing:
`--disable-direct-composition`, `--disable-gpu`, and default rendering all leaked
at the same rate (5–6/6 launches; 15/15 within a session). What looked like a
working fix was single-shot captures landing on an intermittently-excluding state.
Also confirmed: the window/privacy code is byte-identical to v0.7.0 (the version
the user said worked) on the same Electron — so the app never regressed.

Then the user gave the decisive clue: **"working fine at first, but when I move
the window it shows again."** That's the mechanism — on Windows,
`SetWindowDisplayAffinity(WDA_EXCLUDEFROMCAPTURE)` is dropped when the window
enters the modal move/resize loop (a user drag), and **nothing in our code
re-asserted it**: the overlay only saved bounds on move; the main window only
re-applied on `show`. So after any drag the "hidden" window reappeared in capture.

Also learned (dead-ends worth recording): an in-app self-test can't work — a
process's own `desktopCapturer` does NOT see its own content-protected window
(measured: 0 px protected vs 1876 px unprotected), so the app would always read
"hidden"; only a separate process is a valid oracle, and even that is confounded
by the intermittency. So auto-detection was abandoned.

**Fix (`keepContentProtected` in privacy.ts):** re-assert `setContentProtection`
on `move` / `resize` (fire during/after a Windows drag) plus `restore` / `focus`
/ `show`, wired into all three windows (main, overlay, selector). Respects the
on/off state, so it correctly clears protection when Privacy Mode is off.
Removed the false DirectComposition switch. Verified: typecheck · 179 unit
(+4 wiring tests for the re-assertion) · build green. **The OS drag behavior
itself is only reproducible on the affected machine, so the definitive check is
the reporter dragging the real window on this build** — the in-harness real-mouse
drag couldn't be simulated and the machine's WDA state is intermittent.

## The precise trigger: FOCUS / CLICK (not just move) — verified fix

The user refined it further: "when I click the window — even a meaningless
position — it shows suddenly." Focus/activation is the real trigger (move is one
instance of it). This was scriptable, so I reproduced and A/B-verified it:

Magenta-sentinel probe (separate-process capture, the valid oracle):
- protected, unfocused → **0 px** (excluded)
- after focusing the window → **56588 px** (FULLY visible — the bug)
- after re-asserting on focus → **0 px** (hidden again, across 3 cycles)

A/B on the real built app, white backdrop + programmatic clicks:
- **OLD v1.5.1:** before-click lum=140 → after clicks **[140, 21, 21]** (21 = dark
  app UI = LEAK on click)
- **NEW (fix):** before-click lum=140 → after clicks **[140, 140, 140]** (stays
  hidden through every click)

This also explains the earlier "6/6 leak" runs: the E2E harness was focusing the
window. Everything reconciles — a static unfocused protected window is excluded;
**focus/click/move/resize drop the affinity**, and nothing re-asserted it.

Fix = `keepContentProtected` (re-assert on focus/move/resize/restore/show) +
`startContentProtectionWatchdog` (500ms backstop for activation paths that fire no
clean event: taskbar, Alt-Tab, display/DWM changes). Verified end-to-end.

## Final round: real-input leak matrix → the complete, measured picture

User: still flashing "by some event" (and the 500ms watchdog looked like
*interval* flashing, so it was removed in 0482672). Built the definitive
harness: **real OS clicks** (koffi → `mouse_event`/`SetCursorPos`, the same
low-level path as hardware — unlike `win.focus()`, these DO reproduce the
drops) + a **separate-process capture oracle** at ~55fps
(`requestVideoFrameCallback` over a WGC/DXGI `desktopCapturer` stream) with a
white click-through backdrop and per-frame leak metrics, plus positive/negative
controls each run. Scratchpad: `privharness/` (orchestrate.js / oracle-main.js /
input.js).

Measured on this machine (Win 11 26200):

1. `GetWindowDisplayAffinity` **never changes during a leak** (~1M samples,
   stays 0x11 while the window is plainly visible in capture). The drop is
   internal to DWM composition → polling can't detect it; only re-CALLING
   `SetWindowDisplayAffinity` heals it.
2. **Re-calling `setContentProtection` is invisible in capture** (0 leak frames
   across 400+ re-calls, twice) — and WDA never hides a window locally, so
   re-asserting has no visual cost anywhere. The old watchdog wasn't *causing*
   the interval flashing — it was *healing spontaneous drops at a 500ms
   cadence* (each drop sat visible up to 500ms → looked periodic).
3. Real leak classes (old build): **click on an already-active window**
   (~1-in-5, fires NO activation message → stayed visible indefinitely),
   **native `<select>` popups** (separate OS window, affinity 0 → always
   visible while open; matches the user's recording), **z-order changes**
   (`SetWindowPos`, no Electron event → visible for seconds), and **delayed
   spontaneous drops** (~470ms after a drag, no signal at all).
4. `focusable: false` was REJECTED by the data: activation isn't the (only)
   trigger — non-activating clicks drop it too, and would fire even fewer
   healing signals. No UX sacrifice needed (Ask box + dropdowns keep keyboard).
5. `settings.set({ privacyMode })` was never a real API (zod strips unknown
   keys) — the old verify script silently tested nothing. Fixed.

**Fix shipped (this branch):**
- `keepContentProtected`: synchronous re-assert inside
  WM_MOUSEACTIVATE/ACTIVATE/NCACTIVATE/SETFOCUS + **WM_PARENTNOTIFY** (clicks
  on active window) + **WM_WINDOWPOSCHANGED** (z-order) +
  **WM_NCLBUTTONDOWN/ENTER/EXITSIZEMOVE/CAPTURECHANGED** (drag start/end) +
  webContents `input-event` mouseDown, each with collapsed 30/120/300ms
  trailing taps; cleanup on `closed`.
- `startContentProtectionWatchdog(50)`: fast backstop for the spontaneous
  drops (skips when Privacy Mode is off; unref'd).
- All 6 overlay native `<select>`s → new in-window `Dropdown` (ui.tsx): the
  option list renders inside the protected window. (Dashboard pages still use
  native selects — convert opportunistically; during a share the user works
  from the Cue Card.)

**Verified on the fixed build** (46 segments, ~2000 frames, including a REAL
live session via `data.loadSamples` + `session.start`): 8s idle watchdog soak
0/447 frames; click-inactive 5/5, click-active 12/12 (one 1-frame blip),
z-order churn 5/5, drags 8/8, dashboard 3/3 clean; settings + live Type
dropdowns open **zero OS popup windows** and leak nothing; all 4 Format
buttons + toggles clean. Residual: the OS still drops occasionally, now
bounded to ~1–3 capture frames (≤ ~0.1s; ≈0–2 frames of a 30fps Meet share) —
the floor achievable app-side on this Windows build. Definitive sign-off =
user's live Meet/Zoom test.

v1.5.2 changelog rewritten to match reality (earlier text overclaimed).

## Round 2: "continuous clicking still flashes" — opacity red herring + the floor

User: doing things on the Cue Card with continuous click events still flashes.
Built a **storm harness** (real SendInput at ~8 clicks/s for 5s over empty area,
format buttons, dropdown toggling, slider) + a mechanistic isolation probe, and
ran a 4-agent research workflow on the Windows WDA bug. Findings:

1. **Opacity was a RED HERRING.** An early storm run showed the opacity-slider
   segment "leaking" continuously (238/251 frames), so I built an opacity clamp
   (force alpha 255 while Privacy on) + disabled slider. **All of that was
   wrong.** A minimal isolation test (`mini-leak-test.js`: a content-protected
   window on a self-owned backdrop, full z-order control) proved a
   semi-transparent protected window is **still WDA-hidden**: alpha 255 → hidden,
   alpha 178 → hidden, alpha 127 → hidden; only unprotected leaked. Opacity does
   NOT break capture exclusion. **All opacity changes were reverted.**
2. **The "leaks" were a HARNESS ARTIFACT.** Starting a live session calls
   `showOverlay()`/`setAlwaysOnTop`, which reshuffled z-order; my white backdrop
   had been *demoted to the normal band*, so unrelated **background windows rose
   above it** and the oracle captured *those* (a Podia-site email, a
   multi-tenancy chat) — not the Cue Card. Fix: keep the backdrop in the TOPMOST
   `floating` band (above all normal windows, below the `screen-saver` overlay)
   and re-assert it before every measurement. With that, `--live` and the
   slider segment both read **0 leaks**. Lesson: a comparison backdrop must be
   provably just-below the subject and above everything else, or "leaks" are
   just whatever floated up behind it.
3. **The REAL residual** (robust harness, reverted build): during 5s of frantic
   clicking / dragging / dropdown-toggling, **0–1 single 18 ms frame** leaks per
   segment — one capture frame at 55fps, i.e. the drop caught before any heal
   composited. This is the click-storm "flash" the user sees; it is the OS-bug
   floor, not an app defect.

**Round-2 fixes (privacy.ts):** front-loaded the deferred re-assert cascade to
`[4,10,20,40,80,160,300]` ms (the leak window between drop and first heal is
only a frame — hit it early); tightened the watchdog `50 → 20 ms`; added a
**visible-only guard** (skip `!isVisible()` windows — re-asserting on the
deliberately-hidden overlay can leave a blank un-hittable ghost, per Chromium's
own IsVisible guard). Net: click-storm residual from ~2 frames @43ms → ~1 frame
@18ms.

**Research workflow (4 agents, web + source):** confirms this is an
acknowledged, unfixed OS defect (`win32kfull!ChangeWindowTreeProtection`, MS
Q&A #700122) with no KB — only an undistributable `.sdb` shim. The drop is
undetectable (affinity stays 0x11), so mitigation is reactive healing only;
per-drop sub-frame residual is provably unavoidable app-side. The **one
remaining lever** is moving the re-assert OFF the UI thread (a `worker_thread` +
koffi calling `SetWindowDisplayAffinity` every ~5–16ms) so it can't be janked by
main-loop work during rapid interaction — **deferred**: it adds a native
dependency for a sub-frame gain during unrealistic frantic-clicking, and the
research ranked it below the cheap fixes. Recorded as the known next step if the
user still perceives flicker after this. Also flagged for later: audit remaining
separate top-level OS windows (native context menus, date/color pickers,
tooltips) — same unprotectable-popup class as the now-fixed `<select>`; and an
Electron 33 → ≥36 upgrade self-heals the hide/show affinity wipe (but NOT the
interaction drop).

## Round 3: the ACTUAL solution — a non-activating Cue Card (perfect stealth)

User: the watchdog build is "worse than before… feels interval." Decisive
reconciliation of every round: **re-calling `setContentProtection` is itself a
one-frame flicker in a real WGC capture (Zoom/Meet)** — invisible to the
desktop-capturer harness (it samples steady state), plainly visible to the user.
That's why a 20ms watchdog was WORSE than the old 500ms one: 50 re-calls/s = a
constant shimmer vs 2/s. So healing can never be "perfect" — every heal
flickers. The only path to perfect stealth is to **never drop the exclusion, so
we never re-call.**

The drop trigger is window ACTIVATION. So: make the Cue Card **non-activating**
(`focusable: false` → WS_EX_NOACTIVATE). Proved it with the real-input + WGC-ish
oracle harness (`privharness/`), driving the REAL app:
- Non-activating overlay + **BRAINCUE_NO_REASSERT=1 (zero healing)** → **0 leak
  frames** across: 43-click storms, format-button spam, dropdown toggling,
  slider drag, 12× click-active, 5× z-order churn, **8× window-title drag**,
  dropdown open (0 OS popup windows). Full matrix, ~2000 frames, all clean, with
  NO re-assertion at all → nothing to flicker.
- Functional check (real clicks + CDP): format buttons apply, Type dropdown
  opens/selects, **window drags/moves**, and the Ask box types via tap-to-focus
  (ex-style 0x8080108→0x80108 on focus, "hi" received).

**Shipped (this branch):**
- Overlay `focusable: false` ([overlayWindow.ts](../../src/main/windows/overlayWindow.ts)).
- `keepContentProtected(win, { static: true })`: protect once at creation/show,
  **no interaction healing, no timer** — correct precisely because a
  non-activating window can't drop. The default path (event-driven coalesced
  re-assert, no watchdog) stays for the focusable dashboard/selector.
- **The watchdog is gone for good** (it was the interval-flash source).
- Ask-box **tap-to-focus**: `overlay.setKeyboardFocus` IPC →
  `setOverlayKeyboardFocus` toggles `setFocusable` + focus, re-asserting a few
  times to re-hide the brief activation. `onFocus`/`onBlur` on the input.
- 183 unit tests pass (added a `static`-mode test).

**Dead ends buried this round (all were HARNESS ARTIFACTS or wrong theories):**
opacity<1 "leak" (backdrop coverage); "spontaneous ~470ms drop" (fragile
backdrop); the whole heal-faster arms race. The harness backdrop must sit
topmost-floating just under the overlay and be re-asserted per measure, or
background windows masquerade as leaks. And the single-instance lock is
per-app-id: strays from failed runs silently make new launches quit — kill all
electron between runs.

**Known, accepted edge:** typing in the Ask box briefly activates the card
(deliberate, rare during a share); a light re-assert re-hides it. Everything
else is flicker-free.

## Round 4: the ACTUAL root cause — the app's own loopback capture

Everything above (rounds 1–3: activation drops, re-call flicker, `focusable:false`
non-activating card) chased the wrong mechanism. The reporter's precise repro —
"hidden at first; select a profile / **start the interview** → **both** windows
visible and **stay** visible; Google Meet" — pointed elsewhere, and a reliable
probe finally isolated it.

**The probe that cut through the fog:** read each window's real
`GetWindowDisplayAffinity` before/after each step (koffi, cross-process). Unlike
the theorized DWM-internal "drop" (which supposedly kept affinity reading 0x11
and so was undetectable), this flips to a hard **0x0** — a *faithful, reliable*
oracle, unlike the desktopCapturer leak-frame harness that had been misleading me
for three rounds.

**Finding:**
- Boot: overlay + dashboard both `0x11` (hidden). ✓
- `getDisplayMedia` alone: still `0x11`.
- **After `session.start`: both `0x0`.** The exclusion was wiped on BOTH windows.
- `privacy.set(true)` restores `0x11` — but with a **screen** loopback capture
  live, it gets re-wiped within ~1.5s (→ the round-1/2 "interval flicker"); with
  no re-assert it stays visible (→ round-3 "steadily visible").

**Mechanism:** transcribing the interviewer needs system (loopback) audio, which
Chromium only grants alongside a **video** source. That video was
`getSources({types:['screen']})` — a whole-screen capture. While a process
captures the *screen*, Chromium turns OFF `WDA_EXCLUDEFROMCAPTURE` on that
process's own windows (so a self-sharer sees their own windows). So the app was
sabotaging its own stealth for the entire interview. The E2E harness never
reproduced it because it never ran the real `getDisplayMedia`.

**Fix:**
- **Capture a window, not the screen.** New off-screen
  [loopbackAnchor.ts](../../src/main/windows/loopbackAnchor.ts): a tiny,
  content-protected, parked-at-`y:-4000` window; the getDisplayMedia handler
  ([index.ts](../../src/main/index.ts)) returns it as the video source (+
  `audio:'loopback'`), falling back to screen only if it's missing. A **window**
  capture clears the exclusion only ONCE at start (measured), so a re-assert
  sticks — vs a **screen** capture's continuous clearing.
- **Capture shield** ([privacy.ts](../../src/main/services/session/privacy.ts)
  `startCaptureShield`): re-assert `applyContentProtectionToAll` on an 80ms
  interval, **only for the life of a session's capture** (started in
  `sessionManager.goLive`, stopped in `stop`/`shutdown`). This is NOT the old
  always-on watchdog — it runs only during an active capture, where a window
  capture doesn't fight back, so it holds steady instead of flickering.
- **Reverted `focusable:false`** → normal window. It was solving the wrong
  problem and broke Ask-box typing. Removed the `overlay.setKeyboardFocus`
  tap-to-focus IPC/preload/handler and the input's `onFocus`/`onBlur`. Removed
  `keepContentProtected`'s `static` option (nothing uses it).

**Verified (ground-truth affinity, real live session):** windows hidden
**≈99.5%** of an 18s session, worst-case exposure **≤50ms** (one shield
interval), vs **steadily visible** before. Ask box types "hello" even with a
competing foreground window (dashboard) active; overlay ex-style back to
`0x80108` (no `WS_EX_NOACTIVATE`). 184 unit tests pass (dropped the `static`
test; added two shield tests).

**Residual (honest):** not mathematically zero — a rare ≤1-frame clear can slip
through between shield ticks, but at Meet/Zoom screen-share framerates (~5–15fps)
it almost never lands on a captured frame. True zero needs system-audio capture
without any `getDisplayMedia` (a native WASAPI-loopback module) — deferred.

### Round 4b: dashboard native `<select>` popups

The reporter then hit a second, independent leak: **interacting with the
dashboard** (e.g. picking a profile on the Interview page) still showed in the
share. Cause: the shared `Select` ([ui.tsx](../../src/renderer/components/ui.tsx))
was a **native `<select>`**, whose option list opens as a *separate OS popup
window* with display affinity 0 (never covered by the app's exclusion) — the same
class already fixed for the Cue Card. Fix: `Select` now renders the in-window
`Dropdown` under the hood (parses its `<option>` children into options, adapts the
event-style `onChange`), so **all 11 dashboard call sites** are fixed with no
per-site changes. Guarded the Dropdown toggle against a `<label>`/`Field`
re-dispatching the click (double-toggle). Verified with a trusted click on the
Interview page: dropdown opens, 2 options, **zero new OS popup windows**,
selecting updates the value.

### Round 4c: native `title=` tooltips (the dashboard-hover leak)

Reporter's refined repro — "run Google Meet, watch the screen while clicking the
nav and dropdowns here and there" — exposed that the DASHBOARD still leaked on
mere HOVER. A parallel audit (workflow: os-popup, tooltip, lifecycle, WGC, tour
agents) + an empirical harness (`dashscan.js`: drive real hover/click, enumerate
NEW top-level OS windows and read their affinity) pinned it: **native `title=`
tooltips render as a separate top-level OS window (own HWND, affinity 0) not
covered by WDA** — same class as the native `<select>`. `dashscan` caught the
sidebar brand `<Link title="What's new">` (App.tsx:86) spawning a 78×25 affinity-0
window on hover. 21 `title=` sites across the dashboard + ~20 in the Cue Card.

Fix: one global `TooltipShield` ([ui.tsx](../../src/renderer/components/ui.tsx))
mounted once at the app root ([main.tsx](../../src/renderer/main.tsx)) — a
document-level `mouseover` (capture) handler that, on hover, strips the native
`title` (→ `data-tip` + `aria-label`) BEFORE the OS tooltip's ~0.5s delay so the
native tooltip never appears, and renders the text in an in-DOM tooltip covered by
the exclusion. Fixes all current + future `title=` with no per-site edits.
Verified (real SendInput hover, tour disabled): native title stripped, in-window
tooltip shows, ZERO new OS windows; full-dashboard scan across all routes = 0
leaking OS windows.

**Testing gotcha:** the onboarding Tour (gated by `settings.tourDone`,
auto-starts for new users) overlays the UI and blocks hovers → false negatives.
Harnesses must set `api.settings.set({ tourDone: true })` + reload first.

**Still open (audited, NOT yet fixed — separate from the nav/hover repro):** native
Electron dialogs opened from the renderer are separate OS windows not covered by
WDA: `dialog.showMessageBox` (Privacy-off confirm, wipe-all, reset-app),
`dialog.showSaveDialog` (Download PDF), `dialog.showOpenDialog` (resume/JD file
pickers). Only fire on explicit destructive/file actions. Confirms are
convertible to in-window modals; the OS file pickers are inherently un-excludable.

## Round 5: observe-and-heal replaces ALL blind re-asserts + the anchor never worked

User: windows still visible sometimes; the interval `setContentProtection`
(80ms capture shield) "is not good — sometimes it causes the flash". Their
design call, implemented verbatim: **set protection once per window, then run
an observe bot that detects when the OS actually breaks it** — and only then
re-call.

**New machinery:**
- `koffi` is now a production dependency (N-API, no rebuild;
  `electron-builder.yml` asarUnpack added).
- [displayAffinity.ts](../../src/main/services/session/displayAffinity.ts):
  `readWindowAffinity(win)` — the ground-truth `GetWindowDisplayAffinity`
  oracle, in-process. Reads are side-effect-free (unlike re-CALLING
  setContentProtection, which composites a frame that an active WGC capture can
  show — the flash). That asymmetry is the whole design.
- `protectWindow` (privacy.ts): protect at creation + on `show`. That's the ONLY
  proactive protection now. `keepContentProtected`'s hookWindowMessage cascades,
  input-event taps, move/resize/focus handlers: **all deleted**.
- `startProtectionObserver` (privacy.ts): every ~50ms (`BRAINCUE_OBSERVER_MS`)
  read every window's real affinity; if Privacy Mode is on and a window reads
  != 0x11, log a breach + `setContentProtection(true)` — the one situation
  where a re-call can't make anything worse (the window is ALREADY visible in
  the share). Healthy steady state = zero writes = zero flicker. Runs for the
  app's life; `startCaptureShield`/`stopCaptureShield` deleted; capture-start
  blind taps in the getDisplayMedia handler kept only as a fallback when the
  oracle is unavailable.

**Measured (separate-process probe, 15ms sampling — `scripts/privacy-affinity/`):**
- With an **external screen share running** (Google Meet in Chrome — the
  reporter's exact scenario), the OS wipes the affinity on ALL app windows
  every **~5.4s**, app fully idle. Cross-process verified; nothing to do with
  our code. On the old build each wipe left the windows VISIBLE until some
  event-driven re-assert happened to fire ("visible sometimes"), while
  in-session the 80ms shield re-called constantly ("the flash"). The observer
  heals each wipe in ≤62ms with exactly one re-call.
- **The round-4 loopback anchor NEVER worked**: `desktopCapturer.getSources`
  does not enumerate the calling process's OWN windows (isolated in a minimal
  probe: on-screen, off-screen, protected, unprotected — never returned). So
  the title/id match always failed and **every live session silently used
  whole-screen capture** — the continuously-wiping mode — masked by the old
  shield. The `[warn] loopback anchor window not found` line was the tell.
  Fix: hand `getDisplayMedia` the anchor DIRECTLY as a constructed source
  (`{ id: anchor.getMediaSourceId(), name }`) — no enumeration. Verified:
  video track "anchor" + audio track "System audio"; the warn is gone.

**E2E (real driven app, `node scripts/privacy-affinity/drive.js`):** tour
disabled via `settings.set({tourDone:true})`, sample data loaded, every nav page
click-explored, profile picked via the in-window dropdown, REAL interview
started (realtime transcription session opened over the anchor capture), Cue
Card clicked, session stopped. Result: **dashboard 0 capturable samples for the
entire run; overlay 0; anchor one 77ms creation-time dip; 1 observer breach
total** (capture start, healed). 184 unit tests pass (observer + protectWindow
tests replace the cascade/shield tests).

**Field signal (user's `npm run dev` during a live Meet share):** steady stream
of `[privacy] capture protection was wiped … re-protected (now 0x11)` — the
observer visibly detecting + healing the external-share wipe train in real time.

**Residual (honest):** each heal is reactive — a wiped window is capturable for
up to ~one observer tick (≤50ms). At Meet/Zoom share framerates that rarely
lands on a captured frame, but it is not mathematically zero. The old
verify-privacy-capture.mjs (screenshot-based) is superseded by the affinity
harness.

## Round 6 (2026-07-16): move the observer OFF the main thread

Round 5's observer polled `GetWindowDisplayAffinity` on the MAIN thread every
50ms. During a live interview the main process is busy (streaming answers), which
can delay the tick and widen the capturable gap after an external share wipes WDA.
Completed the in-progress migration to an **off-thread worker**
([affinityWorker.ts](../../src/main/services/session/affinityWorker.ts)

and restores any the OS wiped with the **raw `SetWindowDisplayAffinity`** (one DWM

## Round 7 (2026-07-16): the actual root cause of the 0x0 wipes + sub-frame heal

The user still saw intermittent visibility + an interval flash and asked to find
**why the affinity flips to 0x0 at all**. Ran a controlled experiment matrix
(scratchpad `privcheck/`) and nailed it.

**Root cause — an external screen capturer strips WDA; it is NOT our app, NOT
Chromium.** Confirmed by an A/B/A + simultaneous-victim matrix:
- A **minimal Electron window** with zero app logic, protected at creation, is
  wiped to `0x0` on a regular **~5.4s** cadence — but ONLY while an external
  capturer is active.
- A **plain Win32 STATIC window** with the same `WDA_EXCLUDEFROMCAPTURE` and a
  self-heal loop was also in the matrix; the wipe is gated on the capturer, not
  the framework.
- The capturer on this machine is **DeskIn.exe** (a remote-desktop tool the user
  is viewing the session through — `tasklist` shows PID 9100, 397 MB, Console
  session). A remote-desktop/screen-share tool periodically clears
  `WDA_EXCLUDEFROMCAPTURE` on top-level windows so the remote viewer isn't
  looking at black rectangles. When DeskIn's viewer is idle, **zero wipes for
  minutes** on every window type — the wipe perfectly tracks active capture.
- **Disproved** (each by simultaneous A/B, which caught the confound that
  sequential runs hid): `--disable-direct-composition` (still wiped) and
  `--disable-features=CalculateNativeWinOcclusion` (occlusion-off got *more*
  wipes than control in identical conditions — a false positive from DeskIn
  being idle during the earlier sequential run). Neither is the cause.
- Note for the real use case: standard **Meet/Zoom/Teams screen-share** capture
  *respects* WDA and does **not** strip it, so this periodic wipe is largely a
  **DeskIn testing artifact**. The healing still matters for tools that do strip.

**Since we can't stop an external process, minimize the strip→heal exposure (the
flash).** Two measured facts drove the design:
1. Node/libuv `setInterval` on Windows is floored at **~15.6ms** (default timer
   granularity) — even round-6's "12ms" worker actually healed at ~15.6ms.
   `timeBeginPeriod(1)` does NOT tighten libuv timers.
2. `timeBeginPeriod(1)` + a native **`Sleep(1)`** loop DOES run at **~2ms/iter
   at ~0.5% of one core** (measured). Also confirmed a worker thread can call
   `SetWindowDisplayAffinity` on a **main-thread-created** window (affinity is
   not thread-affine).

**Round-7 fix (`affinityWorker.ts`):** the observer worker now raises the timer
resolution (`timeBeginPeriod(1)`) and runs a tight native `Sleep(1)` detect+heal
loop instead of `setInterval`. Because the loop blocks (can't service
`postMessage`), all control state moved to a **`SharedArrayBuffer`** the worker
polls: `Int32[0]` = privacy on/off, `Int32[1]` = HWND count (written last as the
publish barrier), `BigInt64[]` = watched HWNDs. `AffinityObserver.watch/unwatch/
setPrivacy` now write shared memory (no messages); breach counts still flow back
via one throttled `postMessage/sec`. Heal is still the raw
`SetWindowDisplayAffinity`. `privacy.ts` is unchanged behaviorally (docstring +
const-hoist only).

**Measured (deterministic self-strip harness — no DeskIn needed;
`privcheck/healbench3.js`):** the shipped observer as the sole healer, a thread
strips the victim and busy-reads `performance.now()` to time the exact
strip→heal gap over 60 strips → **60/60 healed, median 1.44ms, mean 3.3ms, p95
16ms** (≈4% of one 30fps frame). Down from round-6's ~15.6ms floor and the old
80ms shield. So a DeskIn strip is now exposed ~1.4ms (raw flag flip, no
re-composite) vs up to 50–80ms + a `setContentProtection` re-composite before —
the interval flash should be essentially gone.

**Verified:** **192** unit tests green (rewrote `affinityWorker.test.ts` to
assert the SAB control-block contract); build green; `scripts/privacy-affinity/drive.js`
**PASS** — `protection observer active (off-thread, ~2ms heal)`, real live
session, dashboard + Cue Card never capturable.

**Residual (honest):** still reactive — a strip is capturable for ~1–2ms
(median) up to ~one scheduler hiccup (p95 16ms) before the heal. Far below a
share frame, but not mathematically zero. True zero needs loopback audio without
`getDisplayMedia` (native WASAPI) so nothing ever strips — deferred. The
`timeBeginPeriod(1)` raises this process's timer resolution for the app's life
(minor power cost); acceptable for an active copilot.
